# 客戶定位與產品策略（CUSTOMER POSITIONING）— v3.0

> **本檔是專案戰略軸心**。任何技術決策、Phase 優先級、功能取捨都必須回答：「**這對 50-100 人小型製造業有用嗎？而且能用桌機 Chat 對話完成嗎？**」
> 如果答案是「No」或「不確定」，就應該降級或刪除。

> ⚡ **v3.0 戰略軸轉（2026-05-15）**
> 砍掉 v1/v2 的「LINE Bot + Mobile App + 外協協同」三條支線，全力做「**桌機 Chat 自然語言全 CRUD**」。
> 砍掉的 persona「老吳（外協廠）」於 v3.0 移除（延後到「等客戶反饋」）。

## 1. 目標客戶輪廓（ICP: Ideal Customer Profile）

### 1.1 公司規模

| 維度 | 條件 |
|---|---|
| **員工數** | 50–100 人 |
| **年營收** | 1–5 億新台幣 |
| **產業** | 金屬加工、塑膠射出、PCB、模具、機械零件、3C 配件、食品加工等典型製造業 |
| **生產模式** | OEM / ODM 為主，多為小批量、多品種 |
| **客戶數** | 10–50 家活躍客戶（80/20 法則明顯） |
| **產品數** | 100–500 個 SKU |
| **零件數** | 500–3000 個料號 |
| **廠房** | 1 主廠 + 0–3 個分廠（MESH 多廠用 VMI） |
| **現況** | 仍大量依賴 **Excel + LINE 群組 + 紙本** 管理 |
| **辦公環境** | 每位文職人員 1 台桌機/筆電，Chrome 是主要瀏覽器 |

### 1.2 為什麼這個族群被現有 ERP 拋棄？

| 痛點 | 主流 ERP 的問題 |
|---|---|
| **價格** | SAP/Oracle 動輒新台幣 200 萬起，含顧問費常逾 500 萬 |
| **導入時間** | 6–18 個月，業務不能停 |
| **學習曲線** | 員工需 1–3 個月訓練才上手 |
| **IT 依賴** | 沒有專職 IT，故障無人處理 |
| **流程僵化** | 大廠流程套到小廠，產生不必要的審核關卡 |
| **介面繁雜** | 一個畫面 50 個欄位，員工根本記不住 |
| **客製化貴** | 改一個欄位顧問報價 5–10 萬 |

### 1.3 他們不要什麼（明確 Out of Scope）

- ❌ 複雜的多階審批流程（小廠老闆說了算）
- ❌ 精細的成本會計（多數小廠用「年底盤」）
- ❌ 大型 APS 演算法（GA/SA/TS）→ 老師傅憑經驗排得比演算法快
- ❌ 月結鎖期（會計小姐用 Excel 月底自己對）
- ❌ SOC2 / GDPR 等大企業合規（用不到）
- ❌ 行動 App、LINE Bot、外協協同（**v3.0 移除**，等客戶提需求再說）

## 2. 四個人物誌（Personas，v3.0 全桌機）

### Persona 1｜王董（老闆 / 決策者）

- **年齡**：55–65 歲
- **背景**：技術出身（如機械系），自己創業 20+ 年
- **角色**：簽單、定方向、看大數字
- **典型一天**：早上巡廠 → 中午陪客戶 → 下午對帳 → 晚上回辦公室看當日數字
- **裝置**：辦公室桌機（Chrome 整天開著）
- **核心問題**：
  - 「今天賺了多少？」
  - 「下個月出貨夠嗎？」
  - 「哪個客戶又欠錢？」
- **痛點**：
  - 要等財務、業務各自整理 Excel 才知道狀況
  - 不會用複雜系統，但要全局掌控
  - Excel 報表慢、版本不同步
- **核心 JTBD**：當我**坐在辦公室**，我希望**用一句話問 Chat 就拿到老闆儀表板**，這樣**我能放心**或**及早介入**。
- **理想使用方式**：**桌機 Chat 打字「今天狀況」→ AI 回 5 行摘要 + 圖**

### Persona 2｜小陳（業務）

- **年齡**：28–40 歲
- **背景**：業務出身，常進辦公室 + 出差客戶端
- **角色**：跑客戶、報價、追訂單
- **典型一天**：50% 時間在辦公室處理單，50% 時間出差
- **裝置**：辦公室桌機 + 出差筆電（VPN 回公司）
- **核心問題**：
  - 「客戶要的這個料還有嗎？」
  - 「下週能交多少？」
  - 「上次給 A 客戶的價是多少？」
- **痛點**：
  - 客戶來電要查資料，要切多個系統
  - 報價沒紀錄，下次又重算
  - 客戶下單後不知道何時交貨
- **核心 JTBD**：當我**在客戶電話上**，我希望**打一句話 Chat 就 3 秒查到庫存/價格/交期**，這樣**我能當場成交**。
- **理想使用方式**：**桌機 Chat「客戶 CUST-A001 的 A 產品下批可交多少？」→ AI 立刻回**

### Persona 3｜林廠長（生產主管）

- **年齡**：40–55 歲
- **背景**：現場師傅升上來，技術強但 IT 弱
- **角色**：排工單、調機台、處理異常
- **典型一天**：早會 → 巡線 → 處理插單 → 趕急貨
- **裝置**：辦公室桌機（決策時）+ 廠房大屏（即時看板）
- **核心問題**：
  - 「哪台機台壞了？」
  - 「明天的工單料齊嗎？」
  - 「這張工單能不能拉到 5/20 前完成？」
- **痛點**：
  - 排工單靠經驗 + 黑板（沒紀錄）
  - 缺料才發現要追採購
  - 切多個系統查料 / 機 / 人
- **核心 JTBD**：當我**收到異常**，我希望**桌機 Chat 一句話就釋放/調整工單**，這樣**我能立刻處理**。
- **理想使用方式**：**Chat「把 WO-20260514-002 的工序 30 移到機台 M-05」→ AI 出 ConfirmCard 確認後執行**

### Persona 4｜阿玲（採購兼倉管）

- **年齡**：30–45 歲
- **背景**：行政/會計背景，會用 Excel
- **角色**：開採購單、收料、發料、盤點
- **典型一天**：對帳 → 開 PO → 點貨 → 進系統
- **裝置**：辦公室桌機 + USB 條碼槍（接電腦那種）
- **核心問題**：
  - 「這批料到了沒？」
  - 「哪些料快沒了？」
  - 「上次跟誰買的這料？」
- **痛點**：
  - Excel 不同步（多人改）
  - 開 PO 要切 5 個畫面點到崩潰
  - 安全庫存只能憑記憶
- **核心 JTBD**：當我**收到老闆「補貨」指示**，我希望**Chat 一句話就送出 PO**，這樣**我不用點 5 個畫面**。
- **理想使用方式**：**Chat「跟長江廠下 100 個 M6 螺絲，交期 5/20」→ AI 出 ConfirmCard 列出供應商/數量/金額/交期 → 點確認 → 送出**

## 4. 核心痛點 vs 我們的解法（v3.0）

| # | 客戶痛點 | 傳統 ERP 做法 | 我們的解法 | 對應模組 |
|---|---|---|---|---|
| 1 | **太貴** | 一次性高額授權費 | 開源 + 按需訂閱 + Docker 一鍵部署 | 整個架構 |
| 2 | **太難用** | 要學幾百個畫面 | **桌機 Chat：自然語言對話完成 CRUD** | Multi-Agent LLM + Tool Registry |
| 3 | **點擊累** | 開一張 PO 要點 5 個畫面 | **一句話 + ConfirmCard** | hard-write tool + ConfirmCard |
| 4 | **庫存不同步** | 多人改 Excel | 即時 DB + SSE 推播 + Toast 通知 | Inventory + EventBus |
| 5 | **資料分散** | 訂單/庫存/生產各自 Excel | 單一系統 + 12 domain 整合 | 整個 backend |
| 6 | **老闆要看大數字** | 月底報表 | **即時 Chat 摘要** | Dashboard + Analytics Agent |
| 7 | **缺料才發現** | 人工檢查 | 自動警示 + Toast 即時通知 | ConstraintChecker + SSE |
| 8 | **誤操作恐懼** | 改錯找不回來 | **ConfirmCard + 90s Undo** | RiskTier + audit log |
| 9 | **新人上手慢** | 1-3 個月訓練 | **AI 取代教育訓練（< 2 小時）** | Chat + Workflow guide |
| 10 | **同義詞不通** | 「螺絲 vs M6-BOLT」對不到 | **Glossary + Disambiguation** | Phase 2 |

## 5. 核心承諾（Value Proposition）— v3.0

### 5.1 對王董（老闆）

> 「**打開 Chrome，打一句『今天狀況』，老闆儀表板立刻出現。** 不用學系統、不用裝 App、不用問員工。」

### 5.2 對小陳（業務）

> 「**客戶問什麼，你 3 秒答得出來。** 庫存、交期、歷史價格、應收狀況——對 Chat 說一句話 AI 就告訴你。」

### 5.3 對林廠長（生產）

> 「**異常 Toast 自動跳出，點開 Chat 一句話就調工單。** 缺料、機台異常、進度延遲——不用切 5 個畫面。」

### 5.4 對阿玲（採購）

> 「**開 PO 不用點 5 個畫面。** 對 Chat 說『跟長江廠下 100 個 M6 螺絲』，ConfirmCard 跳出數量/金額/交期，點確認就送出。**改錯有 90 秒 Undo**。」

## 6. 競品定位

| 競品 | 價格 | 優勢 | 劣勢 | 我們的差異化 |
|---|---|---|---|---|
| **SAP Business One** | 200 萬起 | 功能完整 | 太貴、太難 | 1/10 價格、桌機 Chat 全 CRUD |
| **正航 ERP** | 50–100 萬 | 本土化好 | 老介面、點擊累 | 對話式介面 |
| **鼎新 Workflow ERP** | 80–150 萬 | 流程強 | 客製貴、IT 依賴重 | 開源、可自改 |
| **Odoo（開源）** | 0–30 萬 | 開源、模組化 | 中文支援弱、AI 弱 | 中文優先、LLM 內建 |
| **Excel + LINE 群組** | 0 元 | 免費、人人會用 | 不同步、易錯、無 trace | 保留簡單體驗 + 補上 ERP 能力 |

### 6.1 我們的定位（一句話）— v3.0

> **「給 50-100 人廠的對話式 ERP——桌機 Chrome 開著 Chat，用講的就能查/增/改/刪所有單據，價格只要 SAP 的 1/10。」**

### 6.2 競爭優勢護城河

1. **AI 原生架構**：不是事後加 chatbot，是從 day 1 就以 Multi-Agent 設計
2. **MESH 多廠原生**：傳統 ERP 是中央集中式，我們是分散式
3. **桌機 Chat 全 CRUD**：所有競品都是「Chat 只能查、寫入要回畫面」，我們做到 Chat 也能寫入（含確認卡）
4. **ConfirmCard + Undo**：hard-write 操作有人類確認 + 90 秒撤銷，降低誤操作恐懼
5. **開源透明**：客戶可審計、可自改，沒有 vendor lock-in
6. **動態文件治理**：CLAUDE.md + WORKLOG → 系統演進可追溯

## 8. 反向設計：從客戶體驗倒推架構

### 8.1 王董的一天（v3.0 理想情境）

```
07:30  進辦公室開電腦
       │  Chrome 自動開 ERP / chat
       │  打字「早安，今天狀況？」
       │  ↓
       │  AI 回：「早安王董！
       │           ✅ 昨日出貨 12 單（$580K）
       │           ⚠️ 今日有 2 張急單待釋放
       │           🔴 M6 螺絲剩 300 個（低於安全庫存）
       │           已通知阿玲補貨」

10:00  收到 Toast：「⚠️ WO-20260514-002 預計延遲 2 天」
       │  點開 Chat：「為什麼延遲?」
       │  AI 回：「機台 M-05 故障 + 缺 5 個 part」
       │  打字「請廠長處理」→ AI 出 ConfirmCard
       │       「將建立任務指派給 林廠長, 截止今晚 18:00」
       │  點確認 → 任務送出

20:00  在家用筆電 VPN 進來
       │  Chat「今天工廠表現?」
       │  AI 回 5 行摘要 + 一張圖
```

**對應系統功能**：
- Chat 老闆儀表板自動摘要（**Phase 1 強化**）
- Toast 通知（**Phase 3 新增**）
- AI hard-write 任務指派 + ConfirmCard（**Phase 1 重點**）

### 8.2 阿玲的補貨日（v3.0 理想情境）

```
14:00  收到 Toast：「M6 螺絲低於安全庫存」
       │  打開 Chat
       │  打字「跟長江廠下 100 個 M6 螺絲，交期 5/20」
       │  ↓
       │  AI 出 ConfirmCard：
       │   ┌──────────────────────────────┐
       │   │ 確認建立採購單                  │
       │   │ 供應商：長江五金 SUP-002      │
       │   │ 項目：M6 螺絲 × 100           │
       │   │ 單價：$5 × 100 = $500         │
       │   │ 交期：2026-05-20              │
       │   │ [取消]                [確認]  │
       │   └──────────────────────────────┘
       │  按「確認」
       │  ↓
       │  AI 回：「✅ PO-20260515-007 已建立。
       │           系統內部：庫存預留、林廠長收到通知」
       │
14:05  發現交期打錯（要 5/22）
       │  打字「剛剛那筆 PO 改成 5/22」
       │  AI 用 Undo + recreate
```

**對應系統功能**：
- `create_purchase_order` hard-write tool（**Phase 1 Day 3 重點**）
- ConfirmCard schema + 前端組件（**Phase 1 Day 2 重點**）
- 90s Undo（**Phase 2 重點**）

## 9. 給開發者的指引（v3.0）

### 9.1 每寫一行程式碼前先問自己

1. 這個功能 **王董/小陳/林廠長/阿玲** 哪個 persona 會用？
2. 這個功能能 **對 Chat 用一句話講出來** 嗎？（不能就補 tool 描述 / slot）
3. 這個功能是 **hard-write** 嗎？有 ConfirmCard 嗎？
4. 這個功能 **超過 3 步** 才能完成嗎？（如果是，要簡化）
5. 這個功能 **小廠真的需要** 嗎？還是大廠才需要？

### 9.2 黃金原則（v3.0）

> **「能用 Chat 就不要切畫面，能用一句話就不要點 5 下，能自動就不要手動，能砍就砍。」**

### 9.3 取捨優先級

| 取捨情境 | 優先選擇 |
|---|---|
| 功能完整 vs 易用 | **易用** |
| 介面豐富 vs 對話簡潔 | **對話** |
| 流程嚴謹 vs 流程靈活 | **靈活** |
| 高精度演算法 vs 即時回應 | **即時** |
| 大企業合規 vs 中小企業速度 | **速度** |
| 加功能 vs 砍功能 | **砍** |

## 10. 本檔的使用方式

- **新功能提案時**：先檢查是否落在某個 persona 的 JTBD 上

- **每季 review**：與真實客戶訪談更新 persona 與 JTBD
