# 客戶定位與產品策略（CUSTOMER POSITIONING）

> **本檔是專案戰略軸心**。任何技術決策、Phase 優先級、功能取捨都必須回答：「**這對 50-100 人小型製造業有用嗎？**」
> 如果答案是「No」或「不確定」，就應該降級或刪除。

---

## 1. 目標客戶輪廓（ICP: Ideal Customer Profile）

### 1.1 公司規模

| 維度 | 條件 |
|---|---|
| **員工數** | 50–100 人 |
| **年營收** | 1–5 億新台幣（中小企業典型區間） |
| **產業** | 金屬加工、塑膠射出、PCB、模具、機械零件、3C 配件、食品加工等典型製造業 |
| **生產模式** | OEM / ODM 為主，多為小批量、多品種 |
| **客戶數** | 10–50 家活躍客戶（80/20 法則明顯） |
| **產品數** | 100–500 個 SKU |
| **零件數** | 500–3000 個料號 |
| **廠房** | 1 主廠 + 0–3 個外包/分廠（如表面處理、熱處理、檢驗） |
| **現況** | 仍大量依賴 **Excel + LINE 群組 + 紙本** 管理 |

### 1.2 為什麼這個族群被現有 ERP 拋棄？

| 痛點 | 主流 ERP 的問題 |
|---|---|
| **價格** | SAP/Oracle 動輒新台幣 200 萬起，含顧問費常逾 500 萬 |
| **導入時間** | 6–18 個月，業務不能停 |
| **學習曲線** | 員工需 1–3 個月訓練才上手 |
| **IT 依賴** | 沒有專職 IT，故障無人處理 |
| **流程僵化** | 大廠流程套到小廠，產生不必要的審核關卡 |
| **行動支援差** | 多為桌機網頁，老闆/業務在外用不了 |
| **客製化貴** | 改一個欄位顧問報價 5–10 萬 |

### 1.3 他們不要什麼（明確 Out of Scope）

- ❌ 複雜的多階審批流程（小廠老闆說了算）
- ❌ 精細的成本會計（多數小廠用「年底盤」）
- ❌ 大型 APS 演算法（GA/SA/TS）→ 老師傅憑經驗排得比演算法快
- ❌ 月結鎖期（會計小姐用 Excel 月底自己對）
- ❌ SOC2 / GDPR 等大企業合規（用不到）

## 2. 五個人物誌（Personas）

### Persona 1｜王董（老闆 / 決策者）

- **年齡**：55–65 歲
- **背景**：技術出身（如機械系），自己創業 20+ 年
- **角色**：簽單、定方向、看大數字
- **典型一天**：早上巡廠 → 中午陪客戶 → 下午對帳 → 晚上看 LINE
- **裝置**：iPhone（不熟操作但常用 LINE）+ 桌機（極少用）
- **核心問題**：
  - 「今天賺了多少？」
  - 「下個月出貨夠嗎？」
  - 「哪個客戶又欠錢？」
- **痛點**：
  - 要等財務、業務各自整理 Excel 才知道狀況
  - 出差時看不到廠內進度
  - 不會用複雜系統，但要全局掌控
- **核心 JTBD**：當我**在外或下班後**，我希望**用最簡單的方式知道公司今天表現**，這樣**我能放心**或**及早介入**。
- **理想使用方式**：**手機 LINE Bot 問「今天狀況」→ AI 回給我儀表板**

### Persona 2｜小陳（業務）

- **年齡**：28–40 歲
- **背景**：業務出身，每天在客戶端與廠內間奔波
- **角色**：跑客戶、報價、追訂單
- **典型一天**：80% 時間在外，20% 時間回公司
- **裝置**：手機（主要） + 筆電（偶爾）
- **核心問題**：
  - 「客戶要的這個料還有嗎？」
  - 「下週能交多少？」
  - 「上次給 A 客戶的價是多少？」
- **痛點**：
  - 在客戶面前要查資料，要打電話回廠
  - 報價沒紀錄，下次又重算
  - 客戶下單後不知道何時交貨
- **核心 JTBD**：當我**在客戶面前**，我希望**3 秒內查到庫存/價格/交期**，這樣**我能當場成交**。
- **理想使用方式**：**手機 App 用語音問 → AI 立刻回**

### Persona 3｜林廠長（生產主管）

- **年齡**：40–55 歲
- **背景**：現場師傅升上來，技術強但 IT 弱
- **角色**：排工單、調機台、處理異常
- **典型一天**：早會 → 巡線 → 處理插單 → 趕急貨
- **裝置**：手機（常用 LINE 跟班長溝通） + 對講機
- **核心問題**：
  - 「哪台機台壞了？」
  - 「外協廠這批回來了沒？」
  - 「明天的工單料齊嗎？」
- **痛點**：
  - 排工單靠經驗 + 黑板（沒紀錄）
  - 外協廠進度只能打電話問
  - 缺料才發現要追採購
- **核心 JTBD**：當我**在現場巡線**，我希望**手機看就知道哪裡卡關**，這樣**我能立刻處理**。
- **理想使用方式**：**手機儀表板紅綠燈 + 即時推播異常**

### Persona 4｜阿玲（採購兼倉管）

- **年齡**：30–45 歲
- **背景**：行政/會計背景，會用 Excel
- **角色**：開採購單、收料、發料、盤點
- **典型一天**：對帳 → 開 PO → 點貨 → 進系統
- **裝置**：桌機（主要） + 手機（拍照）
- **核心問題**：
  - 「這批料到了沒？」
  - 「哪些料快沒了？」
  - 「上次跟誰買的這料？」
- **痛點**：
  - Excel 不同步（多人改）
  - 點貨拿紙本 → 回辦公室再 key 一次（重複工）
  - 安全庫存只能憑記憶
- **核心 JTBD**：當我**在收貨/盤點**，我希望**手機掃 QR 就更新庫存**，這樣**我不用重 key 一次**。
- **理想使用方式**：**手機掃條碼 + 自動計算 + 警示推播**

### Persona 5｜老吳（資深作業員 / 外協廠老闆）

- **年齡**：50–65 歲
- **背景**：技術老師傅
- **角色**：現場操作 / 外協廠老闆（自己 + 1-2 個徒弟）
- **裝置**：手機（只會用 LINE 跟拍照）
- **核心問題**：
  - 「這批做完了，怎麼回報？」
- **痛點**：
  - 不會用電腦
  - 看不懂英文
  - 不想註冊新帳號
- **核心 JTBD**：當我**完成一批貨**，我希望**用 LINE 拍一下就回報**，這樣**老闆能立刻知道**。
- **理想使用方式**：**LINE Bot + 掃 QR + 語音回報**

## 4. 核心痛點 vs 我們的解法

| # | 客戶痛點 | 傳統 ERP 做法 | 我們的解法 | 對應模組 |
|---|---|---|---|---|
| 1 | **太貴** | 一次性高額授權費 | 開源 + 按需訂閱 + Docker 一鍵部署 | 整個架構 |
| 2 | **太難用** | 要學幾百個畫面 | 自然語言：用講的 / 打的 | Multi-Agent LLM |
| 3 | **行動端弱** | 桌機網頁勉強適配 | **行動優先（Mobile-First）設計** | **Mobile App + LINE Bot** |
| 4 | **庫存不同步** | 多人改 Excel | 即時 DB + SSE 推播 | Inventory + EventBus |
| 5 | **外協看不到** | 打電話/LINE 問 | 外協 QR 派工單 + LINE Bot 回報 | **OutsourceCoordinator** |
| 6 | **資料分散** | 訂單/庫存/生產各自 Excel | 單一系統 + 12 domain 整合 | 整個 backend |
| 7 | **老闆要看大數字** | 月底報表 | **即時儀表板 + AI 摘要** | Dashboard + Chat Agent |
| 8 | **缺料才發現** | 人工檢查 | 自動警示 + 推播 | ConstraintChecker + SSE |
| 9 | **報工要 Key 系統** | 紙本 → 後補 | **手機掃 QR + 語音輸入** | MES + Speech-to-Text |
| 10 | **老師傅不會用** | 直接放棄 | **LINE Bot 跳過註冊** | LINE Webhook |

## 5. 核心承諾（Value Proposition）

### 5.1 對王董（老闆）

> 「**用 LINE 問一句『今天狀況』，老闆儀表板立刻出現在手機上。** 不用裝 App、不用學系統、不用問員工。」

### 5.2 對小陳（業務）

> 「**客戶問什麼，你 3 秒答得出來。** 庫存、交期、歷史價格、應收狀況——說一句話 AI 就告訴你。」

### 5.3 對林廠長（生產）

> 「**手機紅綠燈一目了然。** 缺料、機台異常、外協延遲——系統自動推播，不用一個個問。」

### 5.4 對阿玲（採購）

> 「**收貨/盤點不用再回辦公室 Key 系統。** 手機掃 QR、拍照、語音輸入，就地完成。」

### 5.5 對老吳（外協廠）

> 「**不用註冊、不用學新軟體。** 用 LINE 掃 QR、拍個照、回個訊息——進度自動同步到客戶系統。」

## 6. 競品定位

| 競品 | 價格 | 優勢 | 劣勢 | 我們的差異化 |
|---|---|---|---|---|
| **SAP Business One** | 200 萬起 | 功能完整 | 太貴、太難、行動端弱 | 1/10 價格、自然語言 |
| **正航 ERP** | 50–100 萬 | 本土化好 | 老介面、行動端弱 | 現代 UX、AI 原生 |
| **鼎新 Workflow ERP** | 80–150 萬 | 流程強 | 客製貴、IT 依賴重 | 開源、可自改 |
| **Odoo（開源）** | 0–30 萬 | 開源、模組化 | 中文支援弱、AI 弱 | 中文優先、LLM 內建 |
| **Excel + LINE** | 0 元 | 免費、人人會用 | 不同步、易錯、無 trace | 保留 LINE 體驗 + 補上 ERP 能力 |

### 6.1 我們的定位（一句話）

> **「給 50-100 人廠的 LINE-Native ERP——用 LINE 操作、用手機隨身、用自然語言對話，價格只要 SAP 的 1/10。」**

### 6.2 競爭優勢護城河（為什麼別人複製不了）

1. **AI 原生架構**：不是事後加 chatbot，是從 day 1 就以 Multi-Agent 設計
2. **MESH 多廠原生**：傳統 ERP 是中央集中式，我們是分散式（外廠資料不外流）
3. **LINE-Native**：台灣中小製造業 99% 用 LINE，這是「不離開 LINE 的 ERP」
4. **開源透明**：客戶可審計、可自改，沒有 vendor lock-in
5. **動態文件治理**：CLAUDE.md + WORKLOG → 系統演進可追溯

## 7. 成功指標（KPI）

| 指標 | 量測方式 | 目標 |
|---|---|---|
| **採用率（DAU/MAU）** | 50 人廠中，每天有 30+ 人用過 | 60%+ |
| **AI 對話比例** | 對話操作 / 總操作 | > 50% |
| **手機使用比例** | 手機請求 / 總請求 | > 70% |
| **LINE Bot 訊息量** | 每日 LINE Bot 互動次數 | > 50 次/日/廠 |
| **外協回報率** | 外協工單按時回報 | > 90% |
| **盤點時間縮短** | 人工盤點 vs 掃碼盤點 | 縮短 80% |
| **報表等待時間** | 老闆問問題到拿到答案 | < 10 秒 |
| **導入時間** | 從決定購買到上線 | < 2 週 |
| **教育訓練時間** | 每位員工受訓時數 | < 2 小時 |
| **客單價** | 50 人廠年費（含部署） | 新台幣 30-50 萬 |

## 8. 反向設計：從客戶體驗倒推架構

### 8.1 王董的一天（理想情境）

```
07:00  早餐看 LINE
       │  傳「早安」給 ERP Bot
       │  ↓
       │  Bot 回：「早安王董！
       │           ✅ 昨日出貨 12 單（$580K）
       │           ⚠️ 今日有 2 張急單待釋放
       │           🔴 M6 螺絲剩 300 個（低於安全庫存）
       │           已自動派 Bot 通知阿玲補貨」

10:00  在客戶端聊天，被問「下批可以交多少？」
       │  打開 LINE 問 Bot
       │  ↓
       │  「客戶 CUST-A001 的 A 產品下批可交多少？」
       │  Bot 回：「依目前 MPS，最快可承諾交期 5/20，數量 500 台」

15:00  收到推播：「⚠️ WO-20260514-002 預計延遲 2 天」
       │  點擊 → 進入工單詳情頁
       │  按「請廠長處理」→ 自動 @林廠長 並建立任務

20:00  在家吃飯
       │  問 Bot「今天工廠表現？」
       │  ↓
       │  Bot 回 5 行摘要 + 一張圖
```

**對應系統功能**：
- LINE Bot 整合（**Phase 1 新增**）
- 老闆儀表板自動摘要（**Phase 1 新增**）
- AI 推播決策（基於 EventBus）
- 工單一鍵指派

## 9. 給開發者的指引

### 9.1 每寫一行程式碼前先問自己

1. 這個功能 **王董/小陳/林廠長/阿玲/老吳** 哪個 persona 會用？
2. 這個功能能在 **手機上** 操作嗎？（不能就要重新設計）
3. 這個功能能用 **自然語言** 觸發嗎？（如果是查詢/操作類）
4. 這個功能 **超過 3 步** 才能完成嗎？（如果是，要簡化）
5. 這個功能 **小廠真的需要** 嗎？還是大廠才需要？

### 9.2 黃金原則

> **「能用 LINE 就不要用 App，能用 App 就不要用網頁，能用語音就不要打字，能自動就不要手動。」**

### 9.3 取捨優先級

| 取捨情境 | 優先選擇 |
|---|---|
| 功能完整 vs 易用 | **易用** |
| 桌機體驗 vs 手機體驗 | **手機** |
| 介面豐富 vs 對話簡潔 | **對話** |
| 流程嚴謹 vs 流程靈活 | **靈活** |
| 高精度演算法 vs 即時回應 | **即時** |
| 大企業合規 vs 中小企業速度 | **速度** |
