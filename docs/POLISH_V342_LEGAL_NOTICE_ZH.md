## 1. 使用者帳號管理（R1）⚖️ **資安最敏感**

### 1.1 行為

- LLM 講「給阿明開帳號」→ ConfirmCard → 建立 User row + 連結 Employee
- LLM 講「停用阿明帳號」→ ConfirmCard → User.is_active = False
- **不刪除**資料（保留 audit 軌跡）

### 1.2 法律 / 資安風險

🔴 **權限與責任**：
- 建立超級管理員 = **賦予不受限制之系統存取權**
- 停用帳號 = **立即剝奪該員工存取權**（即使在工作時間中）
- 此操作之**法律意義**等同於：給/收離職員工的辦公室鑰匙

🔴 **未即時停用之風險**：
- 員工離職但帳號未停 → 可能**繼續操作系統**（盜取資料、修改紀錄）
- 屬於《個人資料保護法施行細則》§12「資料安全維護措施」之違反
- ISO 27001 / CMMI 認證**會直接 fail**

🔴 **誤建超級管理員之風險**：
- 一旦給予 `is_superuser=True` → 該帳號**繞過所有 RBAC**
- 包括：刪除老闆的客戶、改自己的薪資（若有 payroll）、清空 audit log
- 取消方式：另一個超級管理員把他降權，但若**只剩這個超管 + 他離職** → **系統失控**

### 1.3 客戶內控義務（**強制**）

- [ ] **限制**`system.config.update` 權限給**老闆 + 1 名 IT 主管**
- [ ] **建立**「新員工帳號開立」**書面審批流程**（HR + 部門主管 + IT 三方核章）
- [ ] **建立**「離職員工帳號停用」**SOP**（離職當日下班前必停）
- [ ] **超級管理員**至少**保留 2 個**（避免單點故障）
- [ ] **每季稽核**未啟用 / 高權限帳號清單

### 1.4 密碼初始化

- 系統強制：≥ 8 字元、含英文 + 數字
- **客戶須告知員工**：**首次登入立即改密碼**
- 老闆**不應**用 LINE / Email 傳明文密碼 → 用**安全管道**（如紙條面交、加密訊息）

### 1.5 Ouvoca 免責

Ouvoca 對因**未即時停用離職員工**所致之資安事件、**誤建超管**所致之系統失控、**密碼傳遞不當**所致之外洩，於適用法律所允許之最大範圍內**不承擔責任**。

## 2. 附件連結到單據（R3）

### 2.1 行為

- 把已上傳的 Attachment 連結到指定 SO / PO / Quote
- 寫入 `Attachment.parsed_target_type` + `parsed_target_id`

### 2.2 著作權風險

- 附件可能含**第三方著作權內容**（廠商寄來的規格書 / 設計圖）
- 客戶**轉發**給其他第三方須**確認授權**
- 若附件含**競爭對手機密**（誤收 email） → 不應留存

### 2.3 個資風險

- 客戶寄來之附件可能含**個資**（合約 / 身分證影本）
- 適用個資法第 27 條「資料安全維護」
- 保存期應符合**法定保存**（商業會計法 §38：5 年）

### 2.4 客戶內控義務

- 附件分類 + 加密儲存
- **不**外流（同 v3.36 §2）
- 定期清理逾期附件

## 3. AI 每人每日上限（R4）

### 3.1 行為

- middleware 攔截 `/api/chat-v2` 等 LLM endpoint
- 預設 **200 次 / day / user**
- 超過 → 回 HTTP 429 + 中文友善訊息
- UTC 00:00 自動重置

### 3.2 限制

- **in-memory counter** — 後端重啟清空
- **單一 worker** — 多 worker 部署不共享（v3.x 改 Redis）
- **per-IP 計算當無 token** — 同 IP 多人共用會誤判

### 3.3 設定方式

`backend/.env` 加：
```
AI_DAILY_LIMIT_PER_USER=500   # 預設 200
```

### 3.4 客戶責任

- **告知員工**此限制（避免員工臨時撞到上限慌張）
- **依據業務需求**調整上限
- 若員工**經常**撞上限 → 可能流程設計不佳（**不是**員工問題）

### 3.5 法律 / SLA 影響

- 上限為**啟發式保護**，**不**取代正式 SLA
- 客戶**不應**僅憑此功能保證「AI 不會燒爆」
- 仍須定期檢視 v3.38 `query_ai_cost_*` 真實用量

## 4. 對話 Transcript 匯出（R6）

### 4.1 行為

- 匯出最近 N 天之 ConversationLog 為 markdown
- 含 user role / assistant role / timestamp / agent

### 4.2 風險

🔴 **transcript 含使用者輸入歷史**，可能含：
- 客戶名 / 料件 / 訂單細節（**商業機密**）
- 員工問句中之**個人意見 / 抱怨**（個資 + 勞動法）
- 密碼錯誤嘗試（雖系統不應 log，但風險仍在）

### 4.3 客戶內控義務

- 匯出後**加密儲存**
- **不**轉發給非授權人員
- 若用於**教育訓練**：**遮罩**客戶 / 員工真名
- 若用於**HR 評估**：**事前告知**員工（涉勞動法）

### 4.4 Ouvoca 免責

Ouvoca 對 transcript 之**外流 / 誤用 / 用於非法目的**，**不承擔責任**。

## 5. 時區設定（R7）

### 5.1 行為

- 設定 Tenant.settings.timezone
- LLM 回答之日期 / 時間自動轉為該時區
- DB 仍存 UTC

### 5.2 限制

- 白名單僅常見時區（Asia/Taipei / Tokyo / NY / London / UTC 等）
- DST（夏令時間）由 Python zoneinfo 處理
- **不影響 DB 之 UTC 儲存**（其他工具仍須自行 format）

### 5.3 風險

- 跨時區交易（如客戶在 EU，廠在 TW）→ **訂單日期可能誤判**
- 客戶應**統一時區策略**

## 6. 全域搜尋（R2）

### 6.1 行為

- 一次搜尋 客戶 + 料件 + 供應商 + 員工
- case-insensitive，每表 N 筆

### 6.2 風險

- 全域搜尋可能讓**低權限員工**看見**不該看到**之資料
- 但**僅顯示基本欄位**（編號 / 名稱），不顯示金額 / 個資詳情
- 完整資料仍須走各別查詢 + 權限檢查

### 6.3 客戶責任

- 限制 `user.profile.read` 不要給匿名 / 臨時人員
- 監控異常頻繁搜尋（可能洩漏前兆）

## 7. 工作天計算（R5）

### 7.1 行為

- 加減 N 個工作天（自動跳過週末 + 台灣 2026 國定假日）
- 內建假日：元旦 / 春節 / 二二八 / 清明 / 勞動 / 端午 / 中秋 / 國慶

### 7.2 限制

- **僅 2026 年**之台灣假日 hard-coded
- 2027 起需**手動更新**（v3.x 路線圖：動態行事曆）
- 不含**公司自訂假日**（如年中、年終休假）
- 不含**特休 / 補休**

### 7.3 風險

- 算錯交期 → 客戶 / 供應商爭議
- 客戶應**人工覆核**重要日期

### 7.4 Ouvoca 免責

工作天計算為**啟發式 helper**；Ouvoca 對因算錯交期所致之**契約違約、罰金、客戶投訴**，**不承擔責任**。

## 8. 案例：常見錯誤使用模式

### 8.1 ❌ 不該做

- 隨意建立超級管理員（沒書面審批）
- 員工離職當天**不**停用帳號
- 把客戶寄來的**競爭對手**附件存進系統
- 用對話 transcript **截圖**員工抱怨給其他主管看
- 把 AI rate limit 上限設極低（< 50）讓員工無法工作

### 8.2 ✅ 應該做

- 開帳號**三方審批**（HR + 部門主管 + IT）
- 離職員工**當日下班前**停用
- 收到陌生 email 附件**先病毒掃描 + 識別來源**才上傳
- 對話 transcript 用於教育**先遮罩**敏感資訊
- AI rate limit 依員工實際業務量設定（建議 200-500）

## 11. 國際法 / 跨國使用警告

| 地區 | 額外法律 | 對 v3.42 影響 |
|---|---|---|
| 🇪🇺 EU | GDPR Art. 17 (刪除權) | 帳號停用不夠 — 員工可要求**真正刪除** |
| 🇺🇸 US | SOX (上市公司) | Audit log 須**不可竄改** — 帳號停用要記 |
| 🇺🇸 US (CA) | CCPA | 員工有「資料攜帶權」— transcript 匯出須支援 |
| 🇨🇳 China | PIPL | 跨境傳輸 transcript 須**單獨同意** |
| 🇯🇵 Japan | APPI | 帳號狀態變更須**留紀錄至少 3 年** |
