# 稅務 / 會計 / 審批模組法律與合規警告（v3.34）

> **本檔性質**：稅務 / 財報 / 審批之**合規提醒**，**不構成稅務、會計、法律意見**。
> 累積適用於 v3.25.10 → v3.34 全部 §6 聲明。
>
> **English version**: [`TAX_ACCOUNTING_LEGAL_NOTICE_EN.md`](./TAX_ACCOUNTING_LEGAL_NOTICE_EN.md)

## ⚠️ 重要：稅務 / 財報為高合規敏感領域

erpilot v3.34 新增之 LLM tools 涉及：
- **電子發票開立 / 作廢**（影響營業稅申報）
- **付款 / 收款記錄**（影響 cash flow、財報）
- **應收 / 應付查詢**（決策依據）
- **多階審批**（影響契約效力）

這些操作**錯誤可能引發**：
- 🔴 稅務檢查（國稅局調帳）
- 🔴 財報不實（會計師簽核責任）
- 🔴 契約爭議（與客戶 / 供應商）
- 🔴 內控違規（SOX / ISO / GMP 認證撤銷）

**請務必依本檔指引使用，並由 CPA / 稅務顧問 / 法務顧問覆核重大操作。**

## 1. 電子發票（issue / void）

### 1.1 開立發票（issue_einvoice_with_confirm）

| 風險點 | 客戶責任 |
|---|---|
| 統編錯誤 | 開立前**必須**用 `validate_tax_id_tool` 驗證；錯誤之 B2B 發票買方無法扣抵 |
| 金額錯誤 | 開立後**錯誤金額**會列入當期申報；事後修改需開折讓單 |
| 上傳財政部 | erpilot **不直接**上傳；客戶需整合「雲端發票平台」/「字軌號管理系統」API |
| 字軌號分配 | erpilot **不管理字軌號**；需依貴司於財政部之配號規則處理 |

### 1.2 作廢發票（void_einvoice_with_confirm）

| 風險點 | 客戶責任 |
|---|---|
| **24 小時限制** | 開立後 24 小時內可作廢；**逾期不可作廢**，需走「折讓單」流程 |
| 雲端通報 | 作廢需通報財政部 / 雲端發票平台；erpilot 之 ConfirmCard **僅標示本系統內狀態** |
| 已扣抵情境 | 若買方已申報扣抵，作廢可能引發複雜對帳；建議事先與買方協調 |

### 1.3 統編驗證（validate_tax_id_tool）

| 限制 | 說明 |
|---|---|
| 僅格式 + checksum | 演算法依台灣國稅局公開規則；**不查核名稱對應**（如「12345678 是否真的是『XX 公司』」）|
| 不查存活 | 不檢查該公司是否歇業 / 暫停營業 |
| 國別僅 TW | 美、中、歐之 VAT/EIN 驗證未實作；其他國別**請使用對應國家之官方驗證** |

### 1.4 月度營業稅（query_monthly_sales_tax）

| 風險 | 說明 |
|---|---|
| **不是 401 表** | 本工具基於 SO 數據概算；**不可直接作為申報依據** |
| 漏算項 | 不含進項稅額、零稅率、免稅銷售、退回折讓等 |
| 帳上 vs 申報 | 本工具讀帳上 SO；申報需含**已開未列**、**列而未開**等調整 |
| **CPA 覆核必要** | 申報前**強制**由 CPA / 記帳業者覆核 |

## 2. 付款 / 收款（payment / receipt）

### 2.1 記錄付款給供應商

| 操作 | 影響 |
|---|---|
| `record_payment_to_supplier_with_confirm` | 建立 **draft JE**：借 AP、貸銀行 |
| 後續 | 必須由會計用 `post_journal_entry_with_confirm` 過帳才入帳 |

**關鍵警示**：
- erpilot 之 JE 為**草稿**；**未過帳不影響財報**
- 過帳後**鎖定**，需走沖正傳票
- 重大金額（如年度超 5% 營收）建議**雙人覆核**（maker-checker）
- 與供應商之**對帳單**需另行比對；erpilot 不替代對帳流程

### 2.2 記錄客戶收款

| 風險點 | 客戶責任 |
|---|---|
| 收款日 vs 開票日 | erpilot 用建單時間；**稅務認列點**可能不同 |
| 部分收款 | 未模型化「分期收款」；客戶若收一半請手動記 |
| 退款 | 不支援 LLM 一鍵退款；需走 JE 沖正 + 銀行作業 |

### 2.3 應收 / 應付查詢

| 工具 | 限制 |
|---|---|
| `query_outstanding_ar` | 僅基於 **AccountsReceivable 表 status=open/partial**；可能漏未建 AR 之賒銷 |
| `query_outstanding_ap` | 推估自 **PO status=received/partial_received**；可能漏直接費用、員工薪資、稅款 |

**重要**：
- 本工具之金額為**指引性**，**非正式財報**
- 月底結帳前應由 CPA 對帳並調整

## 3. 多階審批

### 3.1 批准 / 拒絕之法律效力

`approve_request_with_confirm` 之**法律意義**：
- 批准者之 `employee_id` 寫入 ApprovalStep，**形成審計軌跡**
- 若該批准事後被認定為**疏失**（如批准超額採購），責任歸該員工 / 主管
- erpilot 之 audit log 可作為**內部稽核依據**，**對外法律訴訟**仍需正式書面

### 3.2 拒絕必填原因

`reject_request_with_confirm` **強制** comment 非空：
- 法律 / 內控合規之**對等處理原則**
- 員工被拒提案之**申訴依據**
- 後續若爭議，原因可供追溯

### 3.3 LLM 抽錯 request_id 之風險

老闆說「批准那張 PO」LLM 可能：
- 抽到別張 PO 之審批單
- 未審完即返回（多階情境）

**ConfirmCard 為防線**：使用者按確認前須檢視 request_id 之具體內容（entity_type + amount + initiator）。

## 4. 倉儲揀貨

### 4.1 建立揀貨單

`create_pick_task_with_confirm` 之風險：
- 為**每個 SO item** 建一張 PickTask（可能過多）
- 不檢查**庫存是否足夠**（揀貨時才發現短料）
- 指派之 operator 若無對應權限，揀貨會卡

**建議**：搭配 `query_inventory` 預先確認庫存。

---

## 5. 品保 NCR / CAPA

### 5.1 NCR 之法律性質

`create_ncr_with_confirm` 建立之記錄：
- **不構成**對供應商之**索賠請求**（需另行書面通知）
- **不構成**對員工之**懲戒依據**（需依勞動法規 + 內規）
- **不構成**對客戶之**召回通知**（需法務 + 公關判斷）

### 5.2 CAPA 之合規地位

`create_capa_with_confirm` 之記錄可作為：
- **ISO 9001 / GMP / IATF 16949** 等品質管理體系之矯正措施追蹤
- **客戶稽核**之佐證資料

但**不構成**：
- 對保險公司之**理賠依據**（需獨立鑑定報告）
- 對主管機關之**召回通報**（需依各國法規程序）

## 6. 不擔保條款

於**適用法律所允許之最大範圍內（to the maximum extent permitted by applicable law）**，erpilot 對下列事項**不承擔責任**：

### 稅務面
1. 因 LLM 抽錯 SO 號 / 統編而誤開發票之**稅務罰款**
2. 因作廢逾期 / 漏作廢之**營業稅申報異常**
3. 因 `query_monthly_sales_tax` 數據誤用作申報依據之**逃漏稅風險**
4. 因統編驗證遺漏（如歇業公司）之**買方扣抵爭議**

### 會計面
5. 因 LLM 建之 draft JE 過帳錯誤之**財報失真**
6. 因 AR/AP 查詢遺漏項目所衍生之**現金流誤判**
7. 因付款 / 收款記錄錯誤之**對帳不符**

### 審批面
8. 因 LLM 抽錯 request_id 之**錯誤批准**所衍生之**契約糾紛**
9. 因 ConfirmCard 點錯 / 未細看之**內控失敗**

### 倉儲 / 品保面
10. 因揀貨單建錯之**出貨延遲**
11. 因 NCR / CAPA 記錄不全之**品質認證撤銷**

### 第三方
12. 第三方（客戶、供應商、員工、稅務機關、保險公司、認證機構）依本系統採取行動所衍生之**任何爭議**

## 8. 建議實務做法

### 稅務
- 開立發票前**強制**用 `validate_tax_id_tool` 驗統編
- 每月 5 日前**由 CPA / 記帳業者**用實際 401 表覆核 erpilot 概算數
- 重大發票（如金額 > NT$100 萬）採**雙人開立 + 主管覆核**
- 雲端發票上傳由 IT 整合 API（erpilot 之 mock 不取代）

### 會計
- 重大付款 / 收款（如 > 年營收 5%）必走 **maker-checker**（建單者 ≠ 過帳者）
- 月底 / 季底 / 年底**強制**由 CPA 對帳
- 對外財報依 **GAAP / IFRS** 標準，不可採 erpilot 之內部分析數據

### 審批
- 批准 / 拒絕之**法律證據力**有限；重大決策另留書面 + 簽名
- 拒絕原因之 comment 應**具體**（不只「不同意」），便於日後爭議追溯

### 品保
- NCR / CAPA 記錄為**內部品質管理用**；對外（保險、召回）需獨立程序
- 認證稽核前由品保主管覆核 NCR / CAPA 完整度

## 9. 文化提醒：LLM 不取代專業

erpilot 之承諾：**「自然語言取代教育訓練」**，**不是**「**取代 CPA / 稅務顧問 / 法務**」。

| AI 可以 | 仍需專業判斷 |
|---|---|
| 快速開立發票（依輸入） | 申報策略 / 節稅規劃 |
| 整合 AR/AP 數據 | 月底結帳 / 財報製作 |
| 推薦審批 / 拒絕 | 重大契約 / 法律爭議 |
| 建立 NCR / CAPA | 對外召回 / 保險理賠 |

最終決策應由：
- 業務 / 採購 / 倉管 / 廠長 / 品保 — 依其專業
- 會計 / CPA — 依其專業
- 主管 / 法務 / 稅務顧問 — 依其權限
- 老闆 — 依其角色

**共同承擔。**
