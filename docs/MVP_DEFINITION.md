# MVP 範圍定義（MVP DEFINITION）

> **MVP 哲學**：少做、做精、做對。
> 50-100 人廠最痛的 8 件事 → 做到極致；其餘暫緩。
> 任何超出 MVP 範圍的功能必須有客戶實際反饋才開發。

---

## 2. MVP 不做（Out of Scope）

### 暫緩到 Phase 3+（先看客戶反饋再決定）

| 功能 | 為什麼暫緩 | 復活條件 |
|---|---|---|
| **APS 演算法引擎（GA/SA/TS）** | 50-100 人廠老師傅憑經驗排得比演算法快 | 客戶廠成長到 200+ 人才需要 |
| **S&OP 月度產銷協調** | 小廠通常一週開一次會就夠 | 客戶實際要求才做 |
| **多階審批工作流** | 小廠老闆一聲令下就過了 | 50+ 客戶要求 |
| **複雜成本會計** | 小廠多用「年底盤」模式 | 客戶上市櫃需求 |
| **月結鎖期** | 太重，會計用 Excel 對更快 | 大客戶會計合規需求 |
| **多階 BOM 遞迴展開（>3 階）** | 小廠 BOM 多在 2-3 階以內 | 客戶實際有 4+ 階 BOM |
| **RCCP/CRP 細產能規劃** | 小廠機台少，目視管理夠用 | 客戶要求 OEE 分析 |
| **What-if 多情境模擬** | 小廠憑感覺決策 | 客戶要求科學決策支援 |
| **SOC2 / GDPR 合規** | 小廠用不到 | 接到歐美客戶要求 |
| **演算法選擇器** | 太學術 | 永遠不做（除非變學術專案） |

### 永遠不做（明確刪除）

| 功能 | 原因 |
|---|---|
| **大型企業財務模組（IFRS/合併報表）** | 小廠用不到 |
| **HR 薪資計算** | 一般小廠用 104 人資、出勤打卡機 |
| **CRM 行銷自動化** | 小廠靠老闆人脈 |
| **EDI（電子資料交換）** | 大型供應鏈才用 |
| **MES 全工序自動化** | 小廠多手動 |

## 4. MVP 完成的判準（Definition of Done）

### 4.1 技術判準

- [ ] 50 人廠跑系統 1 個月，**至少 30 人/日**有使用
- [ ] LINE Bot 訊息量 **> 50 次/日**
- [ ] 手機請求 **> 70%** 流量
- [ ] API p95 latency **< 200ms**
- [ ] 0 個 P1 級 bug（系統當機、資料遺失）

### 4.2 商業判準

- [ ] 客戶說「**這比 Excel 方便**」
- [ ] 客戶說「**老闆愛上 LINE Bot**」
- [ ] 客戶說「**外協廠老闆會用**」
- [ ] 客戶願意付 **30-50 萬/年**
- [ ] 客戶願意 **推薦給同業**（NPS > 30）

### 4.3 體驗判準

- [ ] 新員工 **2 小時內** 學會使用
- [ ] 老闆 **不開 App 也能用**（純 LINE）
- [ ] 外協廠老闆 **不註冊也能回報**
- [ ] 報表問題 **10 秒內** 拿到答案

## 6. 給開發者的 MVP 紀律

### 每寫程式前的「五問」

1. **這對 MVP 有貢獻嗎？** 沒有就不寫。
2. **這 50-100 人廠用得到嗎？** 用不到就降級。
3. **這能在手機上操作嗎？** 不能就改設計。
4. **這能用自然語言觸發嗎？** 不能就補 tool。
5. **這超過 3 步操作嗎？** 是就簡化。

### 每完成功能後的「三檢」

1. **能演示給王董看嗎？** 不能就還沒做完。
2. **能在手機上跑得順嗎？** 不能就還沒做完。
3. **能用 LINE 觸發嗎？**（若適用）不能就還沒做完。
